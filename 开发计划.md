# 开发计划

> 状态说明:✅ 已完成 ｜ 🔄 进行中 ｜ 📋 待规划 ｜ ⚠️ 已知问题

本项目为「可视化网页宏录制 + 回放 + 数据提取 + 导出 Excel」桌面工具(MVP 第一版)。
技术栈:Node + TypeScript + Electron + Playwright + exceljs,本地 JSON 存储。

## 模块:内置浏览器与录制

### 核心功能

- ✅ Electron 桌面窗口(地址栏、打开网页、开始/停止录制、运行/保存/加载宏、导出 Excel、日志区)
- ✅ 内置浏览器使用 `<webview>`(Chromium),可浏览任意网页
- ✅ 录制器(`webview-preload.ts`)监听 click / input / press / scroll 并生成步骤
- ✅ 稳定选择器生成(`selector-generator.ts`):data-testid → id → name → aria-label → role+文本 → class → css path → xpath;css path 的每段在可用时携带稳定 class(如 `li.next`),`:nth-of-type` 仅在 class 仍无法区分同标签兄弟时才用,避免「录制后页面增减兄弟元素致选择器命中数变化」
- ✅ click 步骤附语义指纹(`buildFingerprint`:tag/text/aria/href/最近稳定祖先 anchor),随宏 JSON 保存,供回放时主选择器命中 ≠1 通用重定位
- ✅ 录制经 `sendToHost` 回传宿主,渲染层实时累积步骤并展示
- ✅ 导航后自动「重新武装」录制器(保证翻页后操作仍被记录)
- ✅ 新标签页归一:`<webview allowpopups>`(关键前置开关,缺则 Electron 源头丢弃弹窗、handler 不触发)+ 主进程 `did-attach-webview`+`setWindowOpenHandler` 拒绝弹窗并改在同一 webview 内 `loadURL`,录制连续不断、后续操作正常记录;回放端 `context.on('page')` 跟随弹窗切换活动页保持一致
- ✅ 步骤行右键菜单:「标记翻页操作」可将步骤标记为翻页特殊标识并设总页数 N(取消标记/修改页数),标记行带高亮+「翻页 · 共N页」徽标(`pagination`/`pageCount` 字段随宏 JSON 保存)
- ✅ 步骤行右键菜单:「在此后添加滚动到底部」插入 `{type:'scroll-bottom'}` 专用步骤(用于触发无限滚动懒加载),`insertScrollBottom` 仿 `insertPause`;需多次加载就插多条
- ✅ 步骤行右键菜单:「在此后添加等待页面加载完成」插入 `{type:'wait-for-load'}` 专用步骤,`insertWaitForLoad` 仿 `insertScrollBottom`;回放到此等 `load` 事件(整页加载完毕)后再继续
- ✅ 步骤行右键菜单:「修改填写内容」仅对 `fill` 步骤出现,就地编辑 `value`(填写内容)字段,无需重录/改 JSON。`showFillValueInput` 仿 `showPauseReasonInput`(菜单内转文本输入框,预填当前 value,回车/确定生效),与 pause.reason 不同——**不 trim、不删字段、允许空串**(填写内容可能含前后空格且语义上必填);改后 `renderSteps()` 即刷新、点「保存宏」落盘。选择器修正仍由既有「重新点选此步骤的选择器」覆盖
- ✅ 元素拾取器(picker)+「等待元素出现」步骤:步骤行右键「在此后添加等待元素出现(点选)」→ 进入拾取模式,在 `<webview>` 里 hover 高亮 + 点选目标元素 → `webview-preload.ts` 用 `generateSelector` 生成选择器,`sendToHost('picker-result',…)` 回传,renderer 插入 `{type:'waitForSelector',selector}` 步骤(复用既有 `waitForSelector` 类型与回放)。拾取点击被 `preventDefault` 拦截不跳转、`if(picking)return` 不录成 click;拾取期间临时挂起录制、结束恢复;ESC 取消;导航后 `dom-ready` 兜底清理悬挂位置。开关通道 `toggle-picker`(仿 `toggle-recording`)。**已解耦为通用「取选择器+回调」服务**:renderer 侧用 `requestPick(onPicked)` 进入拾取、`pendingPick` 回调槽取代写死的 `pickerInsertAt`,`picker-result` 回来后交给发起方登记的回调(用途与拾取分离),为后续消费者(步骤选择器修正 / AI 规则各 selector 点选填充)铺路;现有「等待元素出现」「等待元素可点击」均是它的消费者,行为零变化
- ✅ 步骤板块 UI 打磨:步骤行 hover 高亮(淡灰底+左侧蓝竖条+平滑过渡),翻页标记行 hover 底色协调加深,翻页徽标升级为渐变 chip(`»` 前缀+阴影+字重),右键菜单项/输入框统一 hover 与 focus 反馈
- ✅ 连续滚动步骤折叠展示(纯前端,`renderer.ts`+`index.html` CSS):无限滚动录制易产生大量连续 `scroll` 步骤淹没视野。`renderSteps()` 由 `forEach` 改 `while` 分组扫描,探测连续 ≥2 条可分组滚动(`isGroupableScroll`:`type==='scroll'&&!pagination`,翻页标记滚动单独成行不折叠),默认只显示组首 + 行尾「连续滚动 N 条 ▾」可点击标识(`.scroll-group-toggle`),点击展开全部子项(`.scroll-group-item` 缩进淡色)/收起。折叠态用 `expandedScrollGroups`(`WeakSet<Step>`,按组首对象引用为锚,非下标,插入/删除/拖拽后仍命中、加载新宏自动 GC)。每行 div 创建逻辑抽 `createStepLine(step,i)`(组首/子项/普通行共用,事件仍绑真实下标,右键/拖拽/徽标零改动)。**实时适配**:分组每次 render 按当前 `steps` 重算,已展开组中间插入新步骤 → 拆成「前段(组首引用不变仍展开)+ 新步骤 + 后段(新组首默认折叠)」,现有 `addStep`/`insertXxx`/`moveStep` 本就调 `renderSteps` 故自动生效。数据(`steps`/`buildMacro`)与回放完全不变
- ✅ 提取规则面板可折叠、默认收起(`.collapsible.collapsed`,点标题 `classList.toggle('collapsed')` 展开/收起,箭头指示状态):该框多由 AI 生成、用户极少手改,收起省空间并防误操作;AI 生成后保持收起不自动展开
- ✅ 右侧各板块统一可收起:在原有「提取规则/插件/浏览器登录态」基础上,补齐「AI 提取(`#ai-panel`/`#ai-title`)」「录制步骤(`#steps-panel`/`#steps-title`)」两块的 `collapsible` 折叠;通用折叠规则补全各 body(`#extract`/`.browser-body`/`.ai-body`/`.plugin-body`/`#steps`)的 `display:none`(顺带修复插件面板原先点标题不真正收起的遗漏);步骤面板收起时 `.steps-panel.collapsed{flex:0 0 auto;min-height:0}` 不再占弹性高度。点各自标题 `classList.toggle('collapsed')`
- ✅ 步骤行拖拽排序:每个 `.step-line` 设 `draggable`,HTML5 拖拽(dragstart/dragover/drop/dragend)上下拖动调整顺序,拖动时半透明 + 蓝色插入指示线;复用 `steps` 数组 + `renderSteps()` 重绘,排序结果随「保存宏」落盘;与右键菜单并存,纯前端(`renderer.ts` 的 `moveStep`)
- ✅ 步骤按「录制来源 URL」分组显示(纯前端 `renderer.ts`+`index.html` + `StepFlags` 加 `recordedUrl?`):步骤常录自不同页面(点链接跳转/翻页/进详情页),平铺看不出归属。**录制时打戳**——`addStep` 对非 goto 步骤用 `safeGetUrl()` 记当前页面 URL 到 `step.recordedUrl`(goto 自带 url),随宏 JSON 保存(同 `pagination` 交集字段,回放忽略),故加载旧宏也能分组。**渲染分块**——`computeStepUrls()` 向前填充算每步 URL(goto.url / recordedUrl 更新"当前 URL",其余继承;旧宏退化为按 goto 边界),`renderSteps()` 按连续同 URL 分成 `.url-group`(可折叠标题:折叠箭头 + 域名+路径截断 + 悬停完整 URL + 「N 步」计数 + body),body 内再做既有连续滚动折叠(原渲染循环抽成 `renderStepRange(container,start,end)`)。折叠态 `collapsedUrlGroups`(`Set<string>` 按 URL 记忆,默认展开,加载宏清空)。**拖拽限制在本组内**——模块级缓存 `stepUrls`(每次 render 刷新、与 steps 对齐),`createStepLine` 的 dragover/drop 加"同组"闸门(`stepUrls[dragFromIndex]!==stepUrls[i]` 则鼠标显禁止 + 拒绝落下),步骤只能在本 URL 组内上下重排、不能跨页出界。回放层/preload/主进程/旁车/undo·autosave 机制零改动(recordedUrl 天然进入 steps 快照/签名)。`npm run build` + `verify-captures`/`verify-merge`/`verify-pagination`(30 行)回归通过

## 模块:回放引擎(MacroRunner)

### 核心功能

- ✅ 读取 JSON 宏,Playwright 启动 Chromium(headed)逐步执行
- ✅ 每种 step 独立处理方法(goto / click / fill / press / scroll / waitForSelector)
- ✅ 每步执行前后打印中文日志(第 N/总 步)
- ✅ 出错截图保存到 `errors/`,返回结构化错误(步骤索引 / 类型 / selector / URL / 原因)
- ✅ 回放全局默认超时(默认 60s,`MACRO_TIMEOUT` 毫秒可覆盖):`page.setDefaultTimeout` / `setDefaultNavigationTimeout`,解决点击后慢页导航等待超时
- ✅ 翻页步骤处理:正常回放跳过被标记的翻页步骤;收集所有翻页步骤构造「翻页序列」(`turnPage` 闭包按文档顺序执行,执行后 `waitForLoadState('domcontentloaded')` 等整页导航稳定再交回提取,防半载页少采/旧分页器误判),总页数取各步 `pageCount` 最大值,传入提取流程驱动翻页
- ✅ click 通用降级:`handleClick` 先 `locator(selector).count()`,唯一命中走主路径;命中 0/多个/非法时若 step 带语义指纹则 `relocateByFingerprint`(anchor 缩小 → 文本精确 → aria → href,取首个可见可用元素)通用重定位;无指纹/重定位失败回退原生 click 沿用出错路径。翻页 click 自动受益,无翻页专用代码
- ✅ 人工介入暂停(`pause` 步骤):回放到该步阻塞,有头浏览器停住等用户在窗口里手动操作(登录/验证码/扫码),点「继续」恢复;依赖倒置——`MacroRunner` 构造注入 `onPause` 回调(core 层不依赖 Electron,无回调默认放行),主进程用 `runId` 隔离的 `resume-macro` 信号恢复;可选 `timeout` 防无人值守永久挂死,超时落入现有出错截图流程;窗口关闭中断回放(`wc.once('destroyed')` reject)
- ✅ 运行中停止/中断回放:工具栏「停止回放」(`#stop-run`)或暂停模态框「停止回放」(`#pause-stop`)可随时中止正在运行的回放。**两层中断**:①协作式取消标志(`MacroRunner.cancelled`,主循环每步前/提取前检查,已置位则抛出跳过后续)保证步骤间干净停;②`cancel()` 主动 `activeContext.close()` 打断正在 await 的 Playwright 操作(慢 waitForSelector/goto/提取),`run()` 从 catch 退出——**关 context 是打断卡住慢步骤的即时性关键**。依赖倒置(同 `onPause`,core 无 Electron 依赖),主进程 `run-macro` 用 `runId` 隔离的 `stop-macro` 信号调 `runner.cancel()`、`macro-run-started` 事件把 runId 推给 renderer。`RunResult.cancelled=true` 标识用户停止(非失败,不弹错误截图);停止=中止并关浏览器(需保留窗口手动操作用「暂停」);暂停中停止=同发 stop+resume(解除 `onPause` 等待后主循环检查 cancelled 抛出)。自检 `scripts/verify-cancel.mjs`
- ✅ 滚动到底部(`scroll-bottom` 步骤):普通顺序步骤,回放到此 `handleScrollBottom` 取页面实际高度 `window.scrollTo(0,document.body.scrollHeight)` 滚到底 + 固定等 1s 懒加载就绪;用于无限滚动页(录不出固定座标),需多次加载就插多条
- ✅ 等待页面加载完成(`wait-for-load` 步骤):普通顺序步骤,回放到此 `handleWaitForLoad` 调 `page.waitForLoadState('load')` 等整页(DOM+所有资源)加载完毕再继续;超时只告警不致命(防轮询型站点 `load` 永不触发挂死),沿用全局默认超时
- ✅ 等待元素可点击(`waitForClickable` 步骤):比 `waitForSelector` 的 visible 更强,要求元素**可交互**(尺寸非零、非 disabled/aria-disabled、视口内时中心点未被遮罩遮挡)。`handleWaitForClickable` 用 `page.waitForFunction` 在页面内做**纯只读判定**、轮询到通过或超时——**不用 trial click**(后者会 scroll into view 改变滚动位置,对本项目 `scroll-bottom`/无限滚动敏感);零副作用(不滚动/不派发事件/不改 DOM)。超时致命(同 waitForSelector 强前置语义);视口外元素降级为只判尺寸+disabled。用于「透明遮罩盖住内容」「disabled 按钮变 enabled」等下一步非点击的同步场景(下一步若是 click,Playwright click 自带 actionability auto-wait 已兜住,无需本步骤)

- ✅ 登录态复用(三条可叠加 UI 开关,默认关,存 `browser-config.json`):①持久化回放 profile——Playwright 改 `launchPersistentContext(userDataDir)`,跨次回放复用同一目录(含 cookie/localStorage),一次登录长期有效;②录制→回放 cookie 注入——回放前从 webview 默认 session 导出 cookies,转 Playwright 格式后 `addCookies` 注入;③录制→回放 localStorage 注入(`injectRecordingLocalStorage`/`#bc-localstorage`)——主进程持有 guest webContents,`executeJavaScript` 读「录制 webview 当前页 origin」的 localStorage,回放端独立 `addInitScript` 按 origin 精准注入(限当前页 origin)。`MacroRunner` 第 4 参 `SessionOptions{userDataDir?,cookies?,localStorage?}`(core 不依赖 Electron,来源/转换在主进程),目录可在侧栏「浏览器登录态」面板勾选/选目录
- ✅ 回放时回填步骤来源 URL(救旧宏,让「按 URL 分组」对无 `recordedUrl` 的旧宏也精确):旧宏录制时未打来源戳,离线只能按 `goto` 边界分组——点链接跳转(无 goto)后的步骤会被错并进前一页组。**回放本就会真实访问每个页面**,故让引擎顺手记录每步 `activePage.url()`(主循环每步执行前 `snapUrl(activePage)` 存 `stepUrls[i]`,语义=该步来源页,与录制打戳一致;`about:blank`/取不到记 null),`RunResult.stepUrls?:(string|null)[]` 随成功/失败(部分执行也回传)带回。renderer 的 `reportRunResult` 调 `backfillRecordedUrls`:长度对齐时把 `stepUrls[i]` 写回**缺失** `recordedUrl` 的非 goto 步骤(只补不覆盖、不动 goto 与新宏已有戳),`renderSteps()` 刷新分组并经既有自动保存落盘。**旧宏跑一次即精确分组、以后打开都对**。纯附带、不改回放行为;`main.ts` 的 `run-macro` 直接透传 result。自检 `scripts/verify-stepurls.mjs`(quotes 首页点作者链接进详情页,断言 click 步来源=首页、其后 waitForSelector 步来源=详情页——正是离线兜底救不了的链接跳转场景)

## 模块:请求拦截改写(录制端 CDP + 回放端 Playwright route,改 POST 请求体)

### 核心功能

- ✅ CDP Fetch 拦截器(`src/main/request-interceptor.ts`,2026-07-09):录制 webview 发出的请求,命中规则的 **POST** 请求按规则改写其 body 顶层字段后放行。**为何用 CDP**:Electron 的 `session.webRequest` 只能改 URL/请求头、**改不了 POST body**(uploadData 只读),要在录制端改请求体唯一路径是 `webContents.debugger` 挂 `Fetch` 域。`attach(wc)`→`debugger.attach('1.3')`(try/catch:DevTools 已开会抢占,记告警不致命)+ 按规则 `urlPattern` `Fetch.enable`(**只暂停命中的 URL**,降开销);`Fetch.requestPaused` 里取完整 body 用 `Fetch.getRequestPostData`(遵守「禁止截断」铁律)、改写后 `Fetch.continueRequest({postData: base64})`(Content-Length 由网络栈重算,同 Puppeteer),**每个暂停请求都必 continue 否则页面卡死**;`detach` 时 `Fetch.disable`+`debugger.detach`+解除文件监听
- ✅ 规则配置 `request-rules.json`(项目根/打包 userData,已 gitignore):`{enabled, rules:[{urlPattern(CDP glob), bodyType?('json'|'form',省略则按 Content-Type/内容嗅探), set?, remove?}]}`;`src/storage/request-rules-store.ts` 加载/校验/首次不存在**自动建 inert 模板**(`enabled:false` + 示例规则,安全且自文档化);**热更新**——`fs.watchFile` 监听文件,改 enabled/规则即时生效无需重启。类型 `RequestRule`/`RequestRulesConfig` 在 `macro-types.ts`
