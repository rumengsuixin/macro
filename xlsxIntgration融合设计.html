🐟macro × xlsxIntgration 融合设计

以 macro 为主体,把 xlsxIntgration 的整合/对账能力做成 macro 的后处理插件(子进程桥接 Python)

日期 2026-07-13 交付形态:设计文档(暂不写代码) 方向:macro 为主体

一、背景与结论

目标:评估本项目 d:\git_object\macro(Electron + TypeScript 的网页录制/回放/数据提取工具)与外部项目 D:\git_object\xlsxIntgration 能否融合。

关键事实:外部项目名叫 xlsxIntgration 但其实是纯 Python 3.8 项目(pandas / openpyxl / xlrd / pdfplumber + websocket-client 手写 CDP),是某公司内部的银行流水整合 + 支付平台对账工具集(7+ 条业务线,输入=文件、输出=xlsx)。两项目语言与运行时完全不同。

✗ 代码级"合并成一个项目"
不可行 —— Node/TS vs Python 3.8,无法函数级复用。
✓ 集成式"融合 / 协作"
可行且有价值 —— 通过子进程 + 文件契约桥接。本文即其落地设计。

二、有融合可能的三点理由

  1. 数据流天然互补衔接。macro 强在"从网页采集 / 触发下载"(通用录制回放 + 请求改写 + 时间线 + 反检测),xlsxIntgration 强在"把下载的银行/支付文件清洗、匹配、对账成统一 xlsx"。前者的输出正是后者的输入。更关键:xlsxIntgration 的代号4/BC/EPIN 三条线正用手写 CDP 驱动 Chrome 逐日下载订单,而这恰是 macro 最成熟的能力,可替代那段脆弱代码。
  2. macro 早已备好"外部能力接入"的热插拔扩展点。registerPostProcessor(吃 ctx.downloads: string[] 文件路径、吐产物文件)+ run-plugin(用户选文件→处理→showItemInFolder 定位)是完整闭环。xlsxIntgration 每条业务线的本质就是"一批文件进、一个 xlsx 出",与后处理器契约几乎完全吻合。
  3. 子进程桥接可做松耦合集成,两边核心都不必重写。macro 主进程是 Node,能 spawn Python 调用 xlsxIntgration 的入口脚本(整合N.py 的 main() -> int),把 macro 采集/下载的文件当输入,Python 产出 xlsx 后返回路径给 macro 定位展示。低侵入、进程隔离、各自演进。

三、诚实说明的限制(设计取舍)

  • 只能走进程 + 文件契约桥接,拿不到函数级复用。
  • xlsxIntgration 高度定制(硬编码银行/平台规则、模板坐标、输入文件严格命名 {公司代号}-{银行}.{ext}),只能作为专用插件接入,不能并入 macro 通用内核。
  • 引入 Python 运行时依赖(需 venv 或 PyInstaller 打包 exe);macro 打包分发时体积增大。
  • xlsxIntgration 只暴露 main()、路径硬编码在 config.py,集成需为它补一层环境变量 / 目录约定。

四、目标架构

macro(Electron / Node / TS)
录制回放 + 下载捕获(list-action → 下载文件)→ runPostProcessors(specs, ctx)
▼ctx.downloads = 文件路径数组
bank-integrate(新增后处理插件)
准备临时输入目录 → 按命名规则放入下载文件 → spawn Python
▼子进程 · env: BANK_INPUT_DIR / BANK_OUTPUT_DIR
xlsxIntgration(Python 3.8)
整合1.py → app.main():读 input → openpyxl 写 → output/*.xlsx;返回退出码 0/非0 + 日志
▼回收产物 xlsx 路径
macro exports/ 定位展示
PostProcessResult{ output } → shell.showItemInFolder

要点:xlsxIntgration 作为一个黑盒 CLI,由 macro 新插件通过子进程调用;输入=一批文件路径,输出=一个 xlsx。前端 UI 零改动(热插拔机制自动收纳新插件)。

五、关键设计

1) 新增后处理插件 bank-integrate(模仿 merge-zip-excel.ts)

  • 新文件 src/core/post-processors/bank-integrate.ts:实现 PostProcessHandler,末尾 registerPostProcessor({type,label,description}, handler) 自注册。
  • 在 src/core/post-processors/index.ts 末尾加 import './bank-integrate';。
  • handler 契约(与既有一致):入 ctx.downloads / exportsDir / stamp + spec.options;出 Promise<PostProcessResult> = { type, output?, message };失败可直接 throw(注册表 catch 成非致命 message)。
  • 多业务线:PoC 先做一个 type(bank-integrate-domestic = 代号1);后续每线一个 type 或复用 spec.options.mode。type 即 UI 列表项,自动上架、支持"勾选随运行"与"直接运行"。

2) 子进程桥接层 python-bridge

  • 新文件 src/core/post-processors/python-bridge.ts,仅用 Node 内置 child_process/fs/path,不引入 Electron。
  • 定位 Python:开发态 = xlsxIntgration 的 venv/Scripts/python.exe;分发态 = PyInstaller 打包 exe。由配置指定。
  • spawn(pythonExe, [entryScript], { cwd: xlsxRoot, env: { …, BANK_INPUT_DIR, BANK_OUTPUT_DIR } });采集 stdout/stderr 转发到 macro 日志;退出码判成败;加超时保护;从输出目录回收 xlsx 拷回 ctx.exportsDir。

3) xlsxIntgration 侧唯一改动 —— config.py 环境变量覆盖(向后兼容)

当前 src/bank_integration/config.py 把 INPUT_DIR / OUTPUT_DIR 硬编码为模块级常量。改为环境变量优先、缺省回退原值:

INPUT_DIR  = Path(os.environ.get("BANK_INPUT_DIR",  DATA_DIR / "input" / "1"))
OUTPUT_DIR = Path(os.environ.get("BANK_OUTPUT_DIR", DATA_DIR / "output"))

不传环境变量时行为完全不变(现有双击启动器、命令行、测试不受影响),template/ 仍用仓库自带模板。若坚持零改动,退而用方案 B(直接把文件拷进 data/input/1、从 data/output/ 取产物),代价是污染固定目录、难并发、需清理 —— 推荐上面的环境变量方案 A。

4) 配置读取 bank-integrate.json(仿 request-rules-store)

  • 新文件 src/storage/bank-integrate-store.ts,读项目根 bank-integrate.json(gitignore)。
  • 字段:pythonExe(可执行/打包 exe)、projectRoot(xlsxIntgration 根 = spawn 的 cwd)、modes(type → entryScript)、naming(下载文件名 → {公司代号}-{银行}.{ext} 映射规则)。
  • 首次运行自动生成 inert 模板(路径留空 + 说明),避免误跑。

5) 文件命名适配(真实摩擦点)

xlsxIntgration 要求输入文件名严格匹配 ^([A-Z]|公司名)-(银行名)\.(xls|xlsx|csv)$(scanner.py + app.py)。macro 下载文件名未必符合。两条路线择一:

  • 简单:约定下载时已产出合规命名文件(文档说明)。
  • 稳妥:插件按 options.naming 规则,把 ctx.downloads 拷入临时输入目录时重命名为合规文件名。
