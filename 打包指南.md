# 打包指南(Windows 安装包,自带浏览器)

> 本文档说明如何把本项目打包成**普通用户双击即装、装上即用**的 Windows 安装包。
> 安装包**自带 Chromium 浏览器**,目标电脑**无需安装 Node、无需安装 Playwright**。

---

## 一、前置条件(三个,缺一不可)

1. **在目标操作系统上打包**
   Playwright 只会下载“当前操作系统”的浏览器,故**打哪个平台的包就在哪个平台上打**:Windows 安装包在 Windows 上打(本节一至九);macOS 版(.dmg/.zip)在 macOS 上打——已配好 GitHub Actions,**无需本地 Mac**(见第十节)。在 Linux 等其它平台上跑打包脚本会被平台校验直接报错阻止。

2. **能联网(仅首次 / 升级 playwright 后)**
   首次打包要下载约 140MB 的 Chromium。断网会失败(脚本会明确报错)。
   之后再打包会**复用上次下载的浏览器,跳过下载**(见下方「打包缓存」),即使断网也能出包。

3. **先装好依赖**
   打包前必须先 `npm install`(否则编译和浏览器下载都跑不起来)。

---

## 二、打包步骤(两条命令)

在项目根目录(`d:\git_object\macro`)打开终端:

```bash
npm install      # ① 安装依赖(只有第一次 / 换电脑 / 依赖变动时需要)
npm run dist     # ② 一键打包
```

`npm run dist` 会自动依次完成:
1. `npm run build` —— 编译 TypeScript + 拷贝静态资源到 `dist/`
2. `npm run pack:browser` —— 准备 Chromium 到 `build/ms-playwright/`(版本未变则复用缓存跳过下载)
3. `electron-builder --win nsis` —— 生成 NSIS 安装包

耗时:首次约 3~6 分钟(含下载);**之后再打包跳过浏览器下载,只需几十秒到一两分钟**。

### 打包缓存(为什么之后不再下载)

`pack:browser` 用 `build/ms-playwright/.pw-version` 记录已下载的 playwright 版本:
- **版本未变 + chrome.exe 在位** → 打印「复用缓存,跳过下载」,直接复用 `build/ms-playwright/`,**0 网络下载**。
- **首次打包 / 升级了 playwright** → 自动清空旧浏览器并重新下载,完成后刷新版本标记(避免旧版本浏览器残留被打进包)。

> `build/ms-playwright/` 已被 `.gitignore` 忽略(不提交),但会**保留在本地磁盘**供下次打包复用。要强制重新下载,删除 `build/ms-playwright/`(或其中的 `.pw-version`)即可。

---

## 三、产物位置

打包完成后,产物在 `release/` 目录:

- **安装包(发给别人的就是这个)**:
  `release\网页宏录制工具 Setup 1.0.0.exe`(约 318MB,已做安装包瘦身,见第十一节)
- 中间产物(免安装版,自己测试可用):
  `release\win-unpacked\网页宏录制工具.exe`

> 版本号 `1.0.0` 来自 `package.json` 的 `version` 字段,改它即可改安装包文件名里的版本号。

## 四、验证(发出去前务必自测一次)

1. **确认浏览器进包了**:打开
   `release\win-unpacked\resources\ms-playwright\chromium-*\chrome-win64\chrome.exe`
   这个文件存在 = 浏览器已随包。

2. **干净环境实测(最重要)**:
   拿一台**没装过 Node、没装过 Playwright** 的 Windows 电脑(或临时把自己电脑的
   `%LOCALAPPDATA%\ms-playwright` 文件夹改个名藏起来),安装该 exe,然后**实跑一次录制 → 回放 → 导出 Excel**。
   能正常打开浏览器、跑完、导出文件,才算真成功。

## 五、安装后,用户数据存在哪

安装版运行时,所有可写数据写到用户目录(不是安装目录,避免权限问题):

```
%APPDATA%\macro-recorder\
├── macros\            # 保存的宏
├── exports\           # 导出的 Excel
├── errors\            # 出错截图
├── browser-profile\   # 持久化登录态目录
├── browser-config.json
└── ai-config.json     # AI 提取配置
```

卸载程序不会自动删这个目录,用户数据可保留。

> `ai-config.json` 在**程序首次启动时自动生成**(写默认 AI 提取配置),路径会同时打到运行日志里;无需手动创建。

## 六、常见问题

| 现象 | 原因 / 解决 |
|---|---|
| 脚本报“仅支持在 Windows 或 macOS 上打包” | 你在 Linux 等其它平台跑了打包脚本;Windows 包用 Windows,macOS 包见第十节 |
| 脚本报“未找到本地 playwright,请先 npm install” | 没装依赖,先 `npm install` |
| 浏览器下载失败 | 没网或被防火墙拦,联网后重试 `npm run dist` |
| 安装包名含中文打包报错(极少见) | 把 `package.json` → `build.productName` 改成英文(如 `MacroRecorder`)再打 |
| 装到目标机回放报 playwright 路径错(极少见) | 在 `package.json` → `build` 里加一行 `"asarUnpack": ["node_modules/playwright/**"]` 再打 |

## 七、原理简述(知其所以然)

- **为什么目标机不用装浏览器**:打包时把 Chromium 放进安装目录的 `resources\ms-playwright`,程序启动时(`src/main/main.ts`)在打包态设环境变量
  `PLAYWRIGHT_BROWSERS_PATH = resources\ms-playwright`,让 Playwright 去**安装目录里**找浏览器,而不是去目标机的系统缓存(那里是空的)。
- **版本为什么不会错配**:打包脚本用项目本地的 playwright 下载 Chromium,二者版本天然一致;运行时也不写死版本号,升级 playwright 后重新打包会自动适配。
- **为什么不每次重下**:`build/ms-playwright/.pw-version` 记录已下版本,与当前 playwright 一致就跳过下载、复用本地缓存;只有首次或版本变更才重下(并清掉旧版本浏览器,防残留打进包)。
- **AI 提取**:安装版默认保持“连不上 Gateway 就提示、不崩溃”。要连局域网 OpenClaw Gateway,在目标机
  `%APPDATA%\macro-recorder\ai-config.json` 的 `openclaw` 段填全连接信息即可,**无需**在目标机铺 `~/.openclaw`(见下一节)。

---

## 八、让同事电脑用上 AI 提取(把连接信息内联到 ai-config.json)

`openclaw` 段支持三项,填全后目标机彻底不依赖 `~/.openclaw`(`src/core/openclaw-client.ts` 会优先用这里的值,缺项才回退读本机 `~/.openclaw`):

```jsonc
"openclaw": {
    "url": "ws://你的局域网IP:18799",      // 必填:默认是 127.0.0.1(同事本机),要改成你 Gateway 所在机器的 IP
    "token": "你Gateway的token",            // 取自你机器 ~/.openclaw/openclaw.json 的 gateway.auth.token
    "identity": {                           // Ed25519 设备身份,取自你机器 ~/.openclaw/identity/device.json
        "deviceId": "...",
        "publicKeyPem": "-----BEGIN PUBLIC KEY-----\n...\n-----END PUBLIC KEY-----\n",
        "privateKeyPem": "-----BEGIN PRIVATE KEY-----\n...\n-----END PRIVATE KEY-----\n"
    }
}
```

操作步骤:

1. **Gateway 端(你的机器)**:确保 Gateway 运行、监听局域网(非仅 `127.0.0.1`)、防火墙放行 `18799`;`ipconfig` 查本机 IPv4。
2. **取值**:从你机器 `~/.openclaw/openclaw.json` 取 `gateway.auth.token`;从 `~/.openclaw/identity/device.json` 整段复制 `deviceId`/`publicKeyPem`/`privateKeyPem`(PEM 里的 `\n` 保持原样,JSON 字符串本就带转义)。
3. **填入**:把上面三项写进同事机 `%APPDATA%\macro-recorder\ai-config.json` 的 `openclaw` 段,`url` 改成你的局域网地址,保存后重开程序。

> ⚠️ **安全提醒**:`identity` 含私钥(`privateKeyPem`),复制给同事 = 同事用你的设备身份连 Gateway。仅在可信同事/内网测试时这么做;更稳妥的做法是为同事单独签发 device 身份并在 Gateway 授权(当前未做自动下发)。

## 九、相关文件(改打包行为时看这里)

- `package.json` → `scripts.dist` / `scripts.dist:mac` / `scripts.pack:browser` / `build`(含 `build.win`/`build.mac`)字段:打包配置
- `scripts/copy-playwright-browser.mjs`:准备并校验浏览器的脚本(平台校验放宽为 win32/darwin、`findChromeBinary` 按平台定位、`.pw-version` 版本标记缓存复用、版本变更清空重下、产物自检)
- `scripts/make-icon.mjs`:生成 `assets/icon.{png,ico,icns}`(纯 Node,无图像依赖;Windows 用 `.ico`、macOS 用 `.icns`)
- `.github/workflows/build-mac.yml`:在 macOS runner 上打 macOS 版(见第十节)
- `src/main/runtime-bootstrap.ts` / `src/main/main.ts` 顶部:打包态设 `PLAYWRIGHT_BROWSERS_PATH=resources/ms-playwright`(用 `process.resourcesPath`,**Win/Mac 通用**)
- `.gitignore`:已忽略 `release/` 和 `build/ms-playwright/`

## 十、macOS 版打包(GitHub Actions,无需本地 Mac)

> 与 Windows 一样自带 Chromium;产物是 **未签名** 的 `.dmg`/`.zip`。因为 Playwright 浏览器不能跨平台下载、PyInstaller 也不能跨平台编译,macOS 版**必须在 macOS 上构建**——已配好 GitHub Actions(macOS runner),推代码即可出包、下载,无需自己有 Mac。

### 1. 触发构建

工作流 `.github/workflows/build-mac.yml`,两种触发:
- **手动**:GitHub 仓库 → Actions → 「Build macOS App」→ Run workflow,可选架构 `arm64`(Apple 芯片,默认)/ `x64`(Intel)。
- **打 tag**:推 `v*` tag(如 `v1.0.0`)自动触发(默认 arm64)。

runner 与架构:`arm64`→`macos-14`(Apple 芯片)、`x64`→`macos-13`(Intel)。**捆绑的 Chromium 与 electron 应用都按 runner 架构产出、单架构**,故要覆盖两种 Mac 就各跑一次(不用 universal——单架构 Chromium 配 universal electron 只在一种架构可用)。

### 2. 下载产物

构建完成后在该 run 的 **Artifacts** 里下载 `macro-mac-arm64`(或 `-x64`),解压得 `.dmg`/`.zip`。

### 3. 未签名包:首次打开绕过 Gatekeeper

本项目无 Apple 开发者账号(CI 里 `CSC_IDENTITY_AUTO_DISCOVERY=false` 跳过签名),Mac 首次打开会提示「无法验证开发者」。任选其一:
- **右键 App →「打开」→ 再点「打开」**(只需一次);或
- 终端 `xattr -cr "/Applications/网页宏录制工具.app"` 去隔离后正常双击。

### 4. 独立工具(银行整合/对账)在 Mac 上对接

独立工具板块调用的是外部 `xlsxIntgration` 的 **Mac 二进制**(与 macro 分开,不打进 macro 包):

1. 在 xlsxIntgration 侧用其 `build-mac.yml`(或 `build_mac.sh`)打出 `bank-integration` 目录(内含 `domestic_bank_integration` 等英文名二进制 + `_internal/`),下载解压。
2. 放到 **`~/bank-integration`**(默认约定路径),或放任意位置后在 `bank-integrate.json` 把各 `modes.<type>.executable` 改成二进制的绝对路径。配置文件在 macro 用户数据目录 `~/Library/Application Support/macro-recorder/bank-integrate.json`(首次运行从产物自带的 Mac 模板拷贝生成)。
3. 给二进制去隔离 + 加执行位:`xattr -cr ~/bank-integration && chmod +x ~/bank-integration/*`。
4. macro 里侧栏「🧰 独立工具」→ 对应工具「运行」→ 选银行文件即可。

> **配置文件从打包时预置的平台专属模板生成(可控)**:仓库 `config-templates/bank-integrate.{win,mac}.json` 纳入 git,由 `package.json` → `build.extraResources` 一并打进产物 `resources/`;首次运行(`bank-integrate.json` 不存在)时,主进程按 `process.platform` 选对应模板拷到 userData,其中 `{HOME}` 占位符展开为用户主目录。**已存在则只读不覆盖**——升级/重装保留用户改动。要调整分发默认路径,改 `config-templates/*.json` 重打包即可(无需改代码)。模板缺失时回退 `src/core/post-processors/bank-integrate-config.ts` 的代码内平台默认。**Windows 与 Mac 配置不通用**(路径/二进制名不同),各平台用各自模板。

## 十一、安装包瘦身(减小分发体积)

打包默认做**三项零风险瘦身**(**Mac + Windows 两平台同时生效**,功能完全不变、业务代码零改动):

| 项 | 做法 | 效果 |
|---|---|---|
| 删无头 shell | `pack:browser` 打包时删掉 `build/ms-playwright/chromium_headless_shell-*` | 未压缩省 ~270MB / 安装包省 ~85MB |
| 裁语言包 | `package.json` → `build.electronLanguages: ["zh-CN","en-US"]` | electron 内置语言包 55 个 → 2 个 |
| 排除 source map | `build.files: ["**/*","!**/*.map","!**/*.d.ts"]` | 剔除 asar 内依赖(exceljs 等)的调试 map/类型定义(~16MB) |

### 为什么删无头 shell 安全

`playwright install chromium` 会装**两份**浏览器——完整版 `chromium-*`(有头)和无头版 `chromium_headless_shell-*`。打包后的应用**默认有头运行**(`src/core/macro-runner.ts` 的 `MACRO_HEADLESS` 未设即 `headless=false`),只用完整版;无头 shell 打进包纯占体积,最终用户一辈子用不到。开发自检走**系统默认缓存**(`%LOCALAPPDATA%\ms-playwright`),与打包用的 `build/ms-playwright` 是两个目录,删后者不影响自检。

### 需要保留无头 shell 时

若要走「手工 `PLAYWRIGHT_BROWSERS_PATH=<repo>/build/ms-playwright MACRO_HEADLESS=1` 跑无头自检」这条 fallback,打包前设环境变量保留即可:

```bash
PACK_KEEP_HEADLESS_SHELL=1 npm run dist        # 保留无头 shell(不瘦身)
```

### 为什么不做「最大压缩」

electron-builder 官方文档明确:`compression:"maximum"` 对已经是压缩态的 Chromium/Electron 二进制「几乎不减小体积,但明显增加构建时间」。故**不启用**,保持默认压缩,换取更快的 CI 构建。

### 实测(2026-07-14,Windows)

安装包 **318MB**,三项优化均验证生效:
- `release\win-unpacked\locales\` 只剩 `zh-CN.pak` + `en-US.pak`;
- `release\win-unpacked\resources\ms-playwright\` 无 `chromium_headless_shell`(只剩 `chromium-*` + `ffmpeg-*` + `winldd-*`);
- `app.asar` 内 `.map` / `.d.ts` 均为 0。

推算优化前约 420MB,省约 100MB(约 24%)。> 注:未打「完全未优化」的 baseline 实测(无头 shell 已删需重下),此对比为基于各删除内容压缩量级的推算;三项优化生效本身是实测确凿的。
