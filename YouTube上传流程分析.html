二、核心结论速览（TL;DR）

上传分三个网络阶段：
  1. ① 会话初始化（start） → POST upload.youtube.com/upload/studio：前端报文件大小 + frontendUploadId，服务端在响应头里返回 scottyResourceId 与上传 URL。
  2. ② 字节上传（upload+finalize） → POST upload.youtube.com/?upload_id=…：把 28MB 视频字节灌入 scotty，耗时 32 秒（真正的时间开销）。
  3. ③ 创建视频（createvideo） → POST /youtubei/v1/upload/createvideo：提交 (frontendUploadId, scottyResourceId) + 元数据，正式生成视频草稿。
两个关键标识的产生与绑定（已逐字比对证实）：
  • frontendUploadId 由前端在 ① 生成；scottyResourceId 由服务端在同一个 ① 的响应里返回 —— 二者在同一次 start 里配对绑定，并绑定了本次文件大小。
  • scottyResourceId 早于字节上传就已分配（它是「资源槽位」句柄，② 只是往槽位灌内容）。
  • createvideo 校验这一对是否同源：只改其中一个必然 INVALID_ARGUMENT，成对复用才有效。

产生时刻① start 请求发出时（t=20.36s）① start 响应返回时（t=20.36s + 853ms）

角色①上传会话标识 ②幂等键（防重复提交）③scotty 资源的配对键指向「已上传字节」的资源句柄，绑定文件大小

5.1 绑定关系（逐字比对证据）

对 ① 与 ③ 逐字比对：
· frontendUploadId：① 请求体值 === ③ 请求体值 → true
· scottyResourceId：① 响应头值（376）=== ③ 请求体 resourceId.scottyResourceId.id（376）→ true
结论：一次 start 产生一对，原封不动带入 createvideo；createvideo 校验二者是否同源。

5.2 frontendUploadId 的「双重身份」（关键约束）

它既是配对键（必须与 scottyResourceId 同源）又是幂等键（两次 createvideo 用相同值会被判「同一次上传的重试」而去重成一个视频）。
⟹ 想创建多个视频就需要多个不同的 frontendUploadId，而每个又必须绑一个同源 scottyResourceId ⟹ 需要「多对」不同的 (fuid, scotty)，而非一对复用多次。

5.3 feedback 轮询与标识的关系

28 次 upload/feedback 的请求体带 continuations: ["mpTPtAK…"]，其 base64 解出含 innertube_studio 字样 —— 即 continuation token 内编码了本次 frontendUploadId，前端据此轮询「这一次上传」的处理/转码进度。

六、Resumable Upload 协议头速查（X-Goog-Upload-*）

头字段方向含义
x-goog-upload-protocol: resumable请求使用可续传协议
x-goog-upload-command: start请求① 发起会话、协商上传地址
x-goog-upload-command: upload, finalize请求② 上传字节并收尾（一次性）
x-goog-upload-header-content-length请求声明文件字节数（start 时即绑定）
x-goog-upload-offset请求本次字节的起始偏移（续传用）
x-goog-upload-header-scotty-resource-id响应服务端分配的资源句柄（= scottyResourceId）
x-goog-upload-url响应② 字节上传的目标地址（含 upload_id）
x-goog-upload-status: active / final响应会话状态：active=待上传 / final=已落库
x-goog-upload-chunk-granularity响应分片粒度（本次 262144 = 256KB）

七、对「复用资源 / 跳过重传」的分析结论

7.1 已证实（可依赖）

  • ✅ body 可改：成对改 (fuid, scotty) 请求仍有效 → createvideo 的安全令牌未对 body 整体签名。
  • ✅ 资源对可跨会话被 createvideo 接受：旧的 (fuid, scotty) 在新会话 createvideo 里返回 200。但注意：createvideo 200 只代表"草稿登记成功"，不代表字节/转码成功（见 §4 阶段③「关键时序」）——最终成败取决于后端异步取 blob 时它是否还在。
  • ✅ 硬约束：frontendUploadId 与 scottyResourceId 必须来自同一次 start（同源配对），只改其一必 400 INVALID_ARGUMENT。

7.2 待验证（决定能否「批量创建多个」）

  • ❓ 同一对能否创建多个：用同一对 (fuid, scotty) 连续 createvideo 两次 —— 出 1 个（被幂等去重）还是 2 个？决定「一对复用多次」是否可行。
  • ❓ 一份内容能否拿多对：多次 start 同一内容能否触发 scotty「秒传」（返回 final 而不重传字节）？本次抓包无 hash、② 完整传了 32s，倾向「不能秒传」，需专门实验。
  • ❓ blob 时效（TTL）：resumable 上传的 blob/upload_id 存活多久？超时后旧对失效。

7.3 三种可能的工作流

模式描述可行性 / 需要
A 反复发同一视频多个视频条目、内容相同取决于 7.2 的「同一对多用」与「秒传」结论
B 批量消费资源对手上攒了多对 (fuid,scotty)，各建一个视频需给 request-rules 加「成对轮换」能力（每次取下一对成对 set）—— 需改代码
C 单个复用每次上传占位/触发后，换成某固定旧资源对现有静态 set 即可（把 frontendUploadId + resourceId 两个字段都写进 set），无需改代码
